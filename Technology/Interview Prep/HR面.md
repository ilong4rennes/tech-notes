## 2. 在你求学这么多年，让你自己受益最大的一种学习方式和方法是什么？ 

我受益最大的方式是"做真实项目倒逼学习"。

**Situation：** 大二刚接触后端开发，理论课学了很多，但遇到真实系统设计的时候完全不知道从哪里下手。

**Task：** 当时我接手 Scotty Labs 的选课平台，要独立负责后端 API 设计和数据库优化，而我对 MongoDB 聚合管道、React Query 这些技术都很陌生。

**Action：** 我没有先去刷教程，而是直接面对需求，遇到不会的就查文档、找 issue、看源码，再对照真实的查询性能数据去验证自己的理解对不对。每次改动我都会实际测查询时间，看数字有没有变化。

**Result：** 最终把查询时间降低了 60%，服务器负载降低了 70%，也真正搞懂了缓存和索引的底层逻辑。

这种方式让我学到的东西留得住，因为每个知识点都有一个真实的上下文和对应的结果。

## 3. 分享一下你自学某一个知识的例子？自学过程中最艰难的地方？ 

我自学 AWS 云架构和生产部署这个过程是比较完整的例子。

**Situation：** DreamScope 这个项目我想做到真正上线，不只是跑在本地。但我之前从没有独立配置过生产级别的云环境。

**Task：** 需要自己搞定 ECS Fargate 部署、ALB 负载均衡、Secrets Manager 凭据注入、S3 加 CloudFront 前端托管，加上断路器和监控。

**Action：** 我从官方文档入手，但 AWS 文档非常分散，一个功能涉及七八个服务，相互依赖关系很难理清。我最难的地方是调通 ECS 任务和 Secrets Manager 的权限链——IAM 权限配置错误的报错信息很模糊，我花了两天反复看日志、逐条对照 IAM policy，才找到问题根源。

**Result：** 最后整套系统成功上线，支持 HTTPS 和 SPA 路由，集成了 Prometheus 指标和结构化日志，可以正常监控运行状态。

最艰难的地方不是技术本身，而是在文档不完整、报错不清晰的时候，怎么保持耐心、系统性地排查问题，这是那次自学给我最大的收获。
## 4. 分享一个你压力很大的故事？最后如何克服的？ 

**Situation：** 去年在迪克体育用品公司实习，实习只有三个月，我负责的是一个从零开始的全栈平台，要支持计算机视觉模型的实时推理可视化，涉及 Flask、Next.js、Docker、LocalStack、S3，对我来说很多技术栈都是新的。

**Task：** 截止到实习最后两周，我需要完成整个平台的功能开发、容器化、测试覆盖，并且要向组里演示。

**Action：** 我把任务拆分，先锁定核心功能的可用性，把边缘情况的测试后置。同时我每天跟 mentor 做一次简短同步，及时暴露阻塞点，避免自己在某个问题上卡太久。压力很大的时候我会有意识地在晚上停下来，保证第二天的状态。

**Result：** 最终实现了 93% 的测试覆盖率，平台顺利演示，模型切换功能被组内认可为一个比较实用的工程设计。

那次经历让我意识到，压力大的时候最重要的不是加班，而是把事情拆小、把沟通频率提高。
## 5. 有没有跨团队合作的经历？遇到意见不一致如何处理？这个过程中你学到了什么？ 

**Situation：** 在迪克体育用品实习时，我们是跨职能团队，PM 和商科背景的业务同事也参与到平台的前期调研里，我负责技术这侧。

**Task：** 调研阶段我们对产品方向有分歧。我倾向于把模型评估的技术指标尽量完整地呈现出来，但业务同事觉得这些数字对他们来说不直观，他们需要的是能快速判断某个模型在具体运动场景下效果好不好的东西。

**Action：** 我们开了几轮讨论，但一直在各说各的。后来我换了个方式，不再推自己的方案，而是先问他们实际用这个平台要完成什么任务、给谁看。他们说主要是拿去跟管理层汇报，需要结论清晰、对比直观。我重新做了一版方案，把技术细节折叠起来，把可视化对比放在最显眼的位置，拿给他们确认，他们也提了几个调整，我们最终一起敲定了这个方向。

**Result：** 平台上线后业务侧反馈上手很快，不需要额外解释。这次合作让我意识到，技术和业务对"好"的定义出发点不同，不是谁说服谁，而是要先找到共同的使用场景，再从那里往回推方案。

## 6. 分享一个就是其他周围的人给到你负面反馈的例子吗？ 

**Situation：** 在迪克实习时，我提了一个 PR，把模型加载逻辑的修改、前端展示层的调整和相关测试都放在了一起。

**Task：** Reviewer 直接告诉我这个 PR 需要拆分，不同的改动混在一起没办法有效 review。

**Action：** 我当时觉得这些改动在逻辑上是连贯的，心里有点不情愿。但我没有解释或者争，而是先问他希望按什么维度来拆。他说最好每个 PR 只做一件事，出问题的时候能快速定位。我理解了之后把它拆成了三个独立的 PR，每个写清楚目的，并且在描述里标注了它们之间的依赖顺序。

**Result：** 那次之后 review 顺畅很多，也没有再出现边界不清楚的问题。我自己也养成了一个习惯，提 PR 之前先问自己这里面有几个独立的改动目的，超过一个就主动拆。

## 7. 在整个 AIGC 浪潮过程中，你错失了哪些机遇？你有做对什么事情，有做错什么事情？ 

做对的事：我比较早开始接触 LLM 的工程落地。DreamScope 这个项目里我集成了 GPT-4，并且做了规范化的 JSON 输出验证和降级处理，不只是调用 API，而是把它嵌进了一个有队列、有缓存、有监控的生产系统里。这让我对 AI 应用的工程复杂性有了实际认知，不只停留在 demo 层面。

错失的机会：我在大模型应用产品层的探索做得不够。周围有同学很早就在做 AI wrapper 产品、做 agent 工具链，我花了更多时间在基础设施和后端上，对 prompt engineering、RAG、fine-tuning 这些方向的实践深度不够。现在回头看，这些是我技能图里比较薄的地方。

做错的事：有一段时间我对 AI 工具的态度是"先看看再说"，觉得很多东西还不成熟。结果是我错过了一些早期踩坑、积累判断力的窗口。现在 AIGC 发展很快，等成熟再学的策略代价其实很高。

## 8. 你认为优秀的工程师所需要具备的三个能力和素质？

我认为是这三个：

第一，**系统性思维**。能在写一个功能之前，先想清楚它在整个系统里的位置，它的上下游依赖是什么，它的失败模式是什么。不只是让代码跑通，而是知道它为什么这样设计。

第二，**在模糊中推进的能力**。真实工程问题很少是题目清晰的。需求不完整、文档缺失、报错信息不透明，这些是常态。优秀的工程师能在信息不足的时候拆解问题、形成假设、验证假设，而不是等到一切都清楚再动手。

第三，**沟通和共享的意识**。技术做得好，但别人看不懂、用不了，价值打折扣。好的工程师会主动让自己的工作对团队透明，无论是代码可读性、文档质量，还是在 code review 和同步会里把背景说清楚。
